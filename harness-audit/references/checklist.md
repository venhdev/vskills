# Harness Engineering Audit — Detailed Checklist

Synthesized from: OpenAI (Codex harness), Anthropic Engineering, LangChain DeepAgents,
Thoughtworks/Birgitta Böckeler, HumanLayer, ignorance.ai, Mitchell Hashimoto.

---

## Category 1: Agent Instruction File (AGENTS.md / CLAUDE.md)

**Purpose:** This file is the backbone of the harness — deterministically injected into
the system prompt every session. Its quality directly impacts agent output.

### ✅ Pass criteria (all 3 required):
- File exists (AGENTS.md, CLAUDE.md, or equivalent)
- Under ~100 lines, hand-written by a human, not auto-generated
- Updated when the agent makes mistakes (signs of iteration: commit history, comments)

### ⚠️ Partial (exists but wrong):
- File too long (>200 lines) — too many non-universal instructions
- Contains directory listings / codebase overviews (agents discover structure on their own)
- Auto-generated by an LLM (ETH Zurich study: LLM-generated AGENTS.md *hurt* performance)
- Full of conditional rules ("if doing X then Y, unless Z")

### ❌ Fail:
- No file exists

### Evidence commands:
```bash
find . -maxdepth 3 -name "CLAUDE.md" -o -name "AGENTS.md" -o -name "AGENT.md"
wc -l AGENTS.md  # check length
git log --oneline AGENTS.md 2>/dev/null | head -5  # iteration history
```

### Anti-patterns (red flags):
- `# Generated by Claude` at the top of the file
- Too many tool-specific instructions (steers the model too hard)
- Hardcoded directory trees inside the file

## Category 2: Context Management

**Purpose:** The context window is the most precious resource. Sub-agents act as a
"context firewall" — parent thread stays clean while sub-agents handle isolated work.

### ✅ Pass criteria:
- Sub-agent or task decomposition pattern exists (agent runs discrete tasks separately)
- Long-running tasks use handoff artifacts (structured JSON/Markdown state files)
- Context reset mechanism exists for multi-session tasks

### ⚠️ Partial:
- Sub-agents mentioned in AGENTS.md but no implementation evidence
- Handoffs are free-form text rather than structured data

### ❌ Fail:
- Everything runs in a single context window with no limits
- No mechanism to prevent context rot

### Evidence commands:
```bash
# Look for handoff / state file patterns
find . -name "*.handoff.*" -o -name "progress.json" -o -name "feature-list.json" \
       -o -name "task-state*" 2>/dev/null

# Look for sub-agent patterns in code
grep -r "sub.agent\|spawn.*agent\|delegate.*agent" . \
     --include="*.py" --include="*.ts" --include="*.js" 2>/dev/null | head -5

# Look for context reset / compaction mentions
grep -ri "context.reset\|context.window\|compaction\|handoff" AGENTS.md CLAUDE.md 2>/dev/null
```

### Key insight (HumanLayer):
> Sub-agents function as a "context firewall" — discrete tasks run in isolated context
> windows so intermediate noise doesn't accumulate in the parent orchestration thread.

## Category 3: Verification Loop

**Purpose:** Agents are biased toward their first plausible solution. The harness must
force a build-verify cycle. LangChain: changing only the harness to add verification
→ +13.7 points on Terminal Bench 2.0.

### ✅ Pass criteria:
- Agent is explicitly instructed: plan → build → verify → fix (no skipping verify)
- Tests run before the agent declares "done"
- Pre-completion checklist or hook equivalent exists (`PreCompletionChecklistMiddleware`)

### ⚠️ Partial:
- "Run tests" mentioned in AGENTS.md but not enforced
- Only happy path tests, no edge case coverage

### ❌ Fail:
- Agent self-reviews its own code and self-approves (no external verification)
- No test infrastructure at all

### Evidence commands:
```bash
# Test infrastructure
find . -name "*.test.*" -o -name "*.spec.*" -o -name "test_*.py" 2>/dev/null | wc -l

# CI pipeline runs tests?
cat .github/workflows/*.yml 2>/dev/null | grep -i "test\|verify\|check" | head -10

# AGENTS.md has verify instruction?
grep -i "verify\|test\|check\|confirm" AGENTS.md CLAUDE.md 2>/dev/null

# Hooks / middleware
find . -name "*hook*" -o -name "*middleware*" -o -name "*checklist*" 2>/dev/null
```

### Key patterns from Anthropic:
- Separate generator agent + evaluator agent (GAN-inspired)
- Evaluator uses Playwright MCP to click through the app like a real user
- Sprint contract: generator + evaluator negotiate the definition of "done" *before* coding

## Category 4: Architectural Constraints

**Purpose:** "Agents are most effective in environments with strict boundaries and
predictable structure." (OpenAI). Constraints become multipliers once encoded into tooling.

### ✅ Pass criteria:
- Linting rules enforced automatically (ESLint, Ruff, Biome, etc.)
- Structural tests or custom linters enforce architectural boundaries
- Module dependency directions are defined and enforced

### ⚠️ Partial:
- Linter exists but doesn't run in CI or pre-commit
- Architectural rules are in docs only, not mechanically enforced

### ❌ Fail:
- No linting at all
- Any pattern allowed, no guardrails

### Evidence commands:
```bash
# Linting config
ls .eslintrc* .flake8 pyproject.toml biome.json ruff.toml 2>/dev/null
cat .pre-commit-config.yaml 2>/dev/null | head -20

# Custom linters
find . -name "*lint*" -o -name "*arch*test*" 2>/dev/null | head -5

# Architectural enforcement
grep -r "ArchUnit\|dependency.check\|forbidden.import\|no-restricted" \
     --include="*.json" --include="*.toml" --include="*.yaml" . 2>/dev/null | head -5
```

### Key insight (OpenAI):
> Custom linter error messages double as remediation instructions — when the agent
> violates an architectural constraint, the error message tells it how to fix it.
> The tooling teaches the agent as it works.

### Key insight (Thoughtworks):
> Greenfield teams can bake harnessability in from day one.
> Legacy teams: retrofitting the harness is harder but more necessary.

## Category 5: Tool & MCP Hygiene

**Purpose:** Too many tools = context bloat = agent enters the "dumb zone" faster.
Every tool description is injected into the system prompt — only load what's actively needed.

### ✅ Pass criteria:
- MCP tool count is controlled (not "connect everything")
- CLI preferred over MCP when the model already knows the CLI (GitHub, Docker, DB)
- Tools documented with usage examples in AGENTS.md (context-efficient response format)

### ⚠️ Partial:
- MCP servers connected but loading all tools without filtering
- Using a full MCP server for what only needs 3-4 tools

### ❌ Fail:
- Multiple unused MCP servers connected in every session
- No tool usage documentation in AGENTS.md

### Evidence commands:
```bash
# MCP configuration
cat .mcp.json 2>/dev/null || find . -name "mcp*.json" 2>/dev/null -exec cat {} \;

# Tool count
grep -c "tools\|server" .mcp.json 2>/dev/null

# CLI wrappers
find . -name "*.sh" -path "*/scripts/*" 2>/dev/null | head -10
```

### Key insight (HumanLayer):
> Writing a small CLI wrapper for the Linear API saves thousands of tokens compared
> to using the full Linear MCP server. Adding 6 usage examples to CLAUDE.md replaces
> the verbose MCP server tool descriptions entirely.

## Category 6: Documentation as System of Record

**Purpose:** Agents only see what lives in the repo. Knowledge in Slack, Google Docs,
or people's heads is invisible to agents. "From the agent's point of view, anything
it can't access in-context while running effectively doesn't exist." (OpenAI)

### ✅ Pass criteria:
- `docs/` directory with structured content (architecture, design decisions, ADRs)
- AGENTS.md acts as a Table of Contents pointing to deeper docs
- Design decisions are versioned in the repo (not just in chat history)

### ⚠️ Partial:
- Docs exist but are outdated or unmaintained
- Good README but no agent-specific context

### ❌ Fail:
- No `docs/` or only a minimal README.md
- Architectural decisions live in Notion, Confluence, or outside the repo

### Evidence commands:
```bash
# Documentation structure
ls -la docs/ 2>/dev/null
find . -name "ADR*" -o -name "*.adr.md" -o -name "ARCHITECTURE.md" 2>/dev/null

# AGENTS.md points to docs?
grep -i "docs/\|see\|refer\|read" AGENTS.md CLAUDE.md 2>/dev/null

# Knowledge freshness
git log --oneline docs/ 2>/dev/null | head -5
```

### Key insight (OpenAI):
> They use AGENTS.md as a ToC (~100 lines), with the real knowledge base in a `docs/`
> directory. A background agent periodically scans for stale docs and opens cleanup PRs.

## Category 7: Observability & Loop Detection

**Purpose:** Traces reveal failure modes → harness improvements. Loop detection
prevents agents getting stuck in "doom loops" (editing the same file 10+ times with
the same broken approach).

### ✅ Pass criteria:
- Tracing/logging exists for agent actions (LangSmith, Braintrust, custom logs)
- Timeout or max-iteration guardrails are configured
- Loop detection pattern implemented (track per-file edit counts or equivalent)

### ⚠️ Partial:
- Basic logging exists but failure modes aren't analyzed
- Timeout configured but no recovery strategy

### ❌ Fail:
- Agent runs with no observability
- No guardrails when the agent gets stuck

### Evidence commands:
```bash
# Observability config
grep -r "langsmith\|braintrust\|trace\|telemetry\|LANGCHAIN_TRACING" \
     --include="*.env*" --include="*.yaml" . 2>/dev/null | head -5

# Max iterations / timeout config
grep -ri "max.iter\|max.turn\|timeout\|loop.detect" \
     AGENTS.md CLAUDE.md *.json *.yaml 2>/dev/null | head -5

# Benchmark tooling
find . -name "harbor*" -o -name "tbench*" 2>/dev/null
```

### Key insight (LangChain):
> `LoopDetectionMiddleware` tracks per-file edit counts. After N edits to the same file,
> inject "consider reconsidering your approach." Simple but effective against doom loops.

## Category 8: Planning & Workflow Design

**Purpose:** "The separation of planning and execution is the single most important
thing I do." (Boris Tane, Cloudflare). Plan mode → execute, never jump straight to code.

### ✅ Pass criteria:
- Agent has a dedicated planning phase (doesn't immediately write code)
- Task decomposition is practiced (feature list, sprint breakdown, task queue)
- "Garbage collection" cycle: periodic check for entropy and inconsistencies

### ⚠️ Partial:
- AGENTS.md mentions planning but no structured workflow
- Decomposition is ad-hoc and inconsistent

### ❌ Fail:
- Agent receives a prompt and codes immediately with no planning step
- No task queue or feature tracking

### Evidence commands:
```bash
# Planning artifacts
find . -name "feature-list*" -o -name "task-queue*" -o -name "sprint*" \
       -o -name "plan.md" 2>/dev/null

# Workflow scripts
find . -path "*/scripts/*" -name "*.sh" -o -path "*/.claude/*" 2>/dev/null

# Initializer / planner agent patterns
grep -ri "planner\|initializer\|decompose\|feature.list" \
     AGENTS.md CLAUDE.md 2>/dev/null
```

### Key insight (Anthropic):
> An initializer agent generates a feature list with 200+ individual items, each with
> explicit test steps, all initially marked "failing." This prevents the agent from
> trying to one-shot the entire project or prematurely declaring victory.

### Key insight (ignorance.ai):
> Planning isn't a single step — it's continuous. Harness building and agent managing
> happen in parallel, each one informing the other.
